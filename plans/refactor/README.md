## Findings considered and rejected (so they aren't re-audited)

- **Split `src/lib/api/http.ts` into auth/cors/rate-limit modules**: rejected —
  280 lines of small focused functions with clear names; splitting adds
  imports without adding clarity. Revisit only if it passes ~450 lines.
- **Generic wallet-resolution / route-handler factory across code+external
  routes**: rejected as over-abstraction — the differences (auth kind, mode
  check, invalid-code rate limiting) are the point of the two route families.
  017 extracts only the identical fragments.
- **Wallet-kind polymorphism (`WalletResolver` interface)**: rejected —
  speculative abstraction for a 2-variant system with no third variant planned.
- **Decompose `fulfillment.ts` into phases**: rejected — 456 lines, cohesive,
  transaction-critical; the risk outweighs the readability gain.
- **Outbound webhooks are "fire-and-forget without retry"** (audit claim):
  **corrected** — `deliverProjectEvent` IS awaited in the webhook route; a
  failure marks the event `error` and returns 500, so the provider retries.
  Residual nuance: a retry after successful fulfillment but failed delivery
  re-runs fulfillment's idempotent path; a durable delivery queue is recorded
  as a direction option below, not a defect.
- **registry.json missing `error.ts` in component items** (audit claim):
  **rejected/false** — components get it transitively via
  `registryDependencies` → pay-provider → pay-client.
- **FormDialog remount-on-open loses half-typed state**: rejected — the
  `key={String(open)}` remount is an intentional action-state reset; losing
  drafts on close is standard dialog behavior.
- **Add memoization across dashboard components**: rejected — no observed
  problem; premature at current list sizes.
- **ConfirmAction vs MutationForm consolidation**: rejected — two patterns
  with a real semantic difference (destructive-confirm vs inline mutation).
- **Provider-service FK missing on `owner_id`** in its schema copy: noted, not
  planned — the web app owns migrations; the service schema is deliberately a
  read-subset. A comment in the service schema would suffice; fold into any
  future schema-touching change.
- **CHANGELOG.md**: process suggestion, maintainer's call; not planned.
- **Structured logger / request IDs across services**: reduced to op-context
  logging in 020; full structured logging deferred until multi-instance
  debugging demands it.
- **Registry `formatCredits` locale prop**: deferred — browser-locale default
  is acceptable; documented in 019's maintenance notes.
- **Pagination for `GET /api/v1/products`** and prior 001-series rejections
  (CORS preflight `*`, seed password, monthly-grant timezone, cnfast): still
  rejected — see `plans/README.md`.

## Direction options (maintainer's call, not planned as work)

- **Durable outbound-webhook delivery queue**: today consumer-endpoint
  delivery rides on provider retry semantics. A `outbound_webhook_deliveries`
  table with backoff + a dashboard replay view would decouple delivery from
  provider retries. Meaningful build; wait for real demand.
- **Generate registry client types from the API** (OpenAPI → `lib/pay` types):
  kills silent client/server type drift while honoring the zero-dependency
  rule (codegen at build time, not runtime deps). Spike-sized.
- **`hosted` billing mode consolidation**: plan 023 gives `PAY_BILLING_MODE` a
  second job (marketing gate). If hosted billing itself stays lightly used,
  consider documenting it as the "official instance" switch generally.
